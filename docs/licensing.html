Activation Path

Trial first, signed activation after payment confirmation.

The desktop app starts with 5 full set exports. After that, the intended production flow is device-bound activation using a payment intent, chain confirmation, and a signed activation token.

Planned production flow

Device IDThe app creates or reads a stable device identifier.
Payment IntentBackend locks a BNB/USD quote for the one-time license.
TransactionUser sends BNB to the configured wallet and submits the tx hash.
WatcherBackend reconciles confirmations and amount before issuing activation.
Signed TokenBackend returns a signed activation token bound to the device.

Operator notes

  • Keep the signing secret off the public surface.
  • Do not expose source code or admin tooling in the public repository.
  • Use separate public and private repositories.
  • Back the activation service with a persistent database before shipping volume.
  • Add replay protection and rate limiting before production traffic.
The public repository should explain the product and workflow. The private repository should keep the code, signing logic, and release engineering.
